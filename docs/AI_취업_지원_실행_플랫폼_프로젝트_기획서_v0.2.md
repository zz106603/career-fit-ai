## 0. 문서 목적과 근거 구분

이 문서는 개발 착수 전에 제품이 해결할 문제, 대상 사용자, MVP 범위와 검증 기준을 합의하기 위한 기획서다. API, 테이블, 클래스, 프롬프트, 세부 Workflow는 이 단계에서 확정하지 않는다.

본문은 다음 표기를 사용해 **첨부자료에서 확인한 사실**과 **범용 제품 요구사항**을 구분한다.

- `[사례 사실]`: 첨부된 이력서·경력기술서·포트폴리오·채용공고에서 직접 확인한 내용
- `[제품 요구]`: 사례를 일반화해 여러 사용자와 직무에 적용할 제품 설계 방향
- `[가설]`: MVP에서 실제 사용자 검증이 필요한 판단

## 1. 프로젝트 배경

취업 준비자는 하나의 지원 여부를 판단하기 위해 여러 문서와 정보를 반복해서 오간다. 자신의 이력서·경력기술서·포트폴리오에서 관련 경험을 찾고, 채용공고의 필수·우대 조건을 분해하며, 회사 공식 자료를 별도로 조사한 뒤, 이를 지원동기와 면접 준비로 연결해야 한다.

범용 생성형 AI를 이용하면 개별 작업은 빠르게 처리할 수 있지만, 대화마다 경력을 다시 입력해야 하고 결과의 근거와 최신성이 일관되게 관리되지 않는다. 특히 사용자가 하지 않은 경험이 자연스럽게 보완되거나, 회사 일반 정보와 최신 공식 발표가 섞이거나, 같은 공고를 다시 분석할 때 판단 기준이 달라질 수 있다.

이 프로젝트는 사용자의 **확정된 경력 정보**를 재사용 가능한 데이터로 만들고, 채용공고 요구사항 및 출처가 있는 회사 정보와 연결하여, 지원 결정부터 지원 준비까지 이어지는 실행 단위를 제공한다.

## 2. 현재 사용자 불편

### 2.1 사용자 정보가 여러 문서에 분산되어 있다

- `[사례 사실]` S1은 대표 경력과 성과를 짧게 요약하고, S2는 같은 프로젝트를 역할·문제·해결·성과로 상세하게 기술하며, S3는 구조도와 개선 전후 설명을 추가한다.
- `[사례 사실]` 예를 들어 신한은행 JV 경험은 S1에서 `RabbitMQ 기반 비동기 처리로 처리율 6배 향상`으로 요약되고, S2·S3에서는 Master/Worker 구조, Impala 조회, OpenFGA 권한 관리, S3·해시 검증 등 세부 근거가 추가된다.
- `[제품 요구]` 서비스는 문서 하나를 정답으로 취급하지 않고, 여러 문서에서 동일 프로젝트의 정보를 병합하되 출처와 상충 여부를 보존해야 한다.

### 2.2 공고마다 표현과 구조가 달라 비교 비용이 크다

- `[사례 사실]` S4는 하나의 공고에 보험계열사, 금융계열사, 제조계열사, 대외 프로젝트 등 여러 S/W 직무가 표 형태로 함께 제시된다. 직무별 업무·지원자격·우대사항·지역이 다르다.
- `[사례 사실]` S5의 한미사이언스 CRM시스템 운영/개발 직무는 CRM 모바일 앱·백엔드 개발 및 운영, RESTful API 설계, 트랜잭션 락·데드락 분석을 업무로 제시하고 Java/Spring Boot 서버 개발 경험을 요구하며 Swagger, Docker/Kubernetes, 제약 CRM·SFA 경험을 우대한다.
- `[제품 요구]` 공고 전체를 단순 요약하는 대신 사용자가 선택한 직무 단위로 주요 업무, 필수 조건, 우대 조건, 경력 요건, 기술, 고용형태, 근무지를 구조화해야 한다.

### 2.3 적합성 판단의 근거가 흩어져 있다

- `[사례 사실]` S1~S3에는 Java·Spring Boot, REST API, RabbitMQ, PostgreSQL, 분산·비동기 처리, 권한 관리 경험이 확인된다. 반면 S5가 우대하는 Swagger 문서 작성이나 제약 CRM·SFA 경험은 첨부자료만으로 확인되지 않는다.
- `[제품 요구]` 요구사항별로 관련 경험을 연결하고 `충족 / 부분 충족 / 확인 불가 / 미충족`을 구분해야 한다. 키워드가 없다는 이유만으로 곧바로 미충족으로 단정해서는 안 된다.

### 2.4 회사 조사와 지원동기 작성이 분리되어 있다

- 사용자는 공고 분석 후 다시 공식 홈페이지, 채용 페이지, 기술 블로그, 최근 공식 발표를 조사해야 한다.
- 수집한 회사 정보가 왜 해당 사용자와 직무에 중요한지 연결하지 않으면 지원동기에 활용하기 어렵다.
- `[제품 요구]` 회사 사실, 출처, 발행일 또는 확인일을 저장하고, 사용자 경험과 연결 가능한 근거만 지원 준비에 제안해야 한다.

### 2.5 지원 진행 상태와 판단 이력이 남지 않는다

- 관심 공고, 검토 결과, 보완할 사항, 지원 여부가 브라우저 탭·메모·대화에 분산된다.
- `[제품 요구]` 공고별 분석 결과와 근거를 저장하고 `관심 / 지원 검토 / 지원 예정 / 지원 완료 / 보류 / 지원하지 않음` 상태를 관리해야 한다.

## 3. 해결하려는 핵심 문제

### 핵심 문제 정의

> 사용자가 확인한 경력 정보를 기준으로 채용공고 요구사항과 최신 공식 회사 정보를 근거 있게 연결하여, “지원할 것인가”와 “무엇을 근거로 어떻게 준비할 것인가”를 일관된 형태로 결정하지 못하는 문제를 해결한다.

이를 네 가지 하위 문제로 나눈다.

1. 비정형 경력 문서에서 재사용 가능한 경험 단위를 만들기 어렵다.
2. 공고 요구사항과 사용자 경험을 일관된 기준으로 비교하기 어렵다.
3. 회사 정보의 출처·시점·직무 관련성을 함께 관리하기 어렵다.
4. 분석 결과가 실제 지원 상태 관리와 준비 행동으로 이어지지 않는다.

이 제품은 채용 합격을 예측하거나 자기소개서를 대신 제출하는 서비스가 아니다. **근거를 정리하고 의사결정과 준비 행동을 돕는 서비스**다.

## 4. 목표 사용자

### 4.1 1차 목표 사용자

다음 특성을 가진 **IT·개발 직군의 경력직 또는 중고신입 취업 준비자**다.

- 이력서 외에 경력기술서·포트폴리오 등 복수 문서를 보유한다.
- 한 경험이 여러 문서에 서로 다른 수준으로 기술되어 있다.
- 여러 공고에 반복 지원하며 매번 관련 경험을 다시 찾아야 한다.
- 지원동기 작성 전 회사 공식 정보 조사에 시간을 많이 쓴다.
- AI의 빠른 정리는 원하지만 허위 경험 생성과 불명확한 출처는 원하지 않는다.

`[사례 사실]` S1~S3의 사용자는 2021.08~2023.10 백엔드 경력과 이후 개인·팀 프로젝트가 있고, 의료·금융·데이터 플랫폼 경험을 서로 다른 문서 형식으로 보유한다. 이를 1차 MVP의 대표 페르소나로 사용한다.

`[제품 요구]` 초기 검증 데이터와 요구사항 분류는 IT·개발 직군에 맞추되, 경력·프로젝트·역할·성과·공고 요구사항·근거를 저장하는 핵심 구조에는 특정 기술명이나 개발 직군 전용 필드를 고정하지 않는다. 장기적으로 다른 직무의 경험 및 요구사항 분류 체계를 추가할 수 있는 범용 구조를 유지한다.

### 4.2 후순위 사용자

- 신입이지만 학교·교육·대외활동·프로젝트 경험이 여러 문서에 분산된 사용자
- 기획, 데이터, 인프라, 보안, 영업, 연구 등 공고 요구사항을 경험 근거와 연결할 필요가 있는 사용자
- 단일 회사의 여러 직무를 비교해야 하는 사용자

### 4.3 MVP 비목표 사용자

- 대규모 인재 검색과 후보자 평가가 필요한 기업 채용 담당자
- 자동 지원·대량 지원을 원하는 사용자
- 실시간 채용공고 집계 사이트를 원하는 사용자

## 5. 핵심 사용자 시나리오

### 시나리오 A: 경력 기준정보 만들기

1. 사용자가 이력서 또는 경력기술서를 등록한다.
2. AI가 경력, 프로젝트, 기술, 역할, 문제 상황, 해결 행동, 성과를 추출 후보로 제시한다.
3. 사용자는 원문 근거를 보며 잘못된 병합, 수치, 기간, 역할을 수정한다.
4. 사용자가 확정한 항목만 이후 공고 분석의 근거로 사용된다.

`[사례 적용]` S1~S3에서 정밀 의료 빅데이터 플랫폼과 신한은행 JV 프로젝트를 동일 경험으로 연결하되, `처리율 6배`, `조회 속도 2배`, `운영 효율 50%` 같은 수치는 원문 위치와 함께 제시하고 사용자 확정을 받아야 한다.

### 시나리오 B: 공고 분석과 지원 판단

1. 사용자가 공고 본문을 입력하고, 여러 직무가 있으면 분석 대상을 선택한다.
2. AI가 직무 단위 요구사항과 근무 조건을 구조화한다.
3. 시스템이 확정 경력에서 요구사항별 관련 경험을 검색한다.
4. AI가 검색된 근거 안에서 충족 상태와 이유를 제안한다.
5. 사용자는 강점, 부족한 역량, 확인할 사항과 지원 판단을 검토한다.

`[사례 적용]` S5 CRM 직무의 `Java/Spring Boot 기반 서버 개발 경험`은 S1~S3의 여러 프로젝트와 연결될 수 있다. `제약 CRM·SFA 경험`은 첨부자료에서 확인되지 않으므로 확인 불가 또는 미충족 후보로 제시하되 사용자가 별도 경험을 추가할 수 있어야 한다.

### 시나리오 C: 회사 근거와 지원 준비

1. 사용자가 회사 조사 실행을 요청한다.
2. Workflow가 회사 공식 홈페이지의 회사 소개, 공식 채용 페이지, 공식 기술 블로그 또는 뉴스룸·보도자료·IR 등 최근 공식 자료만을 검색한다.
3. 정보마다 출처, 발행일 또는 확인일, 정보 시점을 기록한다.
4. 일반 회사 정보, 최신 정보, 공고 시점 정보를 구분한다.
5. 공고 1건당 최종 채택 자료는 회사 소개·채용 페이지·최근 공식 자료를 모두 합쳐 최대 5건으로 제한한다.
6. AI가 직무 특성과 사용자 경험을 연결할 수 있는 지원 근거와 확인 질문을 제안한다.

### 시나리오 D: 자기소개서 문항 준비(Should)

핵심 폐쇄 루프를 검증한 뒤 구현하는 후순위 시나리오다.

1. 공고 본문에 자기소개서 문항이 있으면 문항을 분리한다.
2. 문항 유형을 경험형, 지원동기형, 문제해결형 등으로 분류한다.
3. 경험형 문항에는 확정된 사용자 경험을, 회사·직무형 문항에는 출처가 있는 회사 정보를 추천한다.
4. MVP는 완성 답안 자동 제출이 아니라 근거 카드와 작성 방향 제공에 집중한다.

### 시나리오 E: 지원 상태 관리

1. 사용자가 분석 결과를 저장하고 지원 상태를 변경한다.
2. 상태별 공고 목록과 마지막 분석 시점을 확인한다.
3. 사용자는 보류·지원하지 않음의 이유와 이후 할 일을 기록한다.

## 6. 서비스가 제공하는 가치

| 가치 | 사용자에게 제공되는 결과 |
|---|---|
| 경력의 재사용 | 문서마다 흩어진 경험을 한 번 확인한 뒤 여러 공고에 재사용 |
| 근거 중심 판단 | 요구사항별 상태, 관련 경험, 원문 근거를 함께 확인 |
| 최신성 관리 | 회사 정보에 출처와 시점을 붙이고 일반·최신·공고 시점 정보 구분 |
| 허위 생성 억제 | 확정 경력 밖의 경험은 생성하지 않고 확인 불가를 명시 |
| 실행 가능성 | 강점 요약에 그치지 않고 부족한 정보, 준비 행동, 문항별 근거 제안 |
| 지원 이력 축적 | 분석 결과와 지원 상태를 저장해 반복 조사와 판단 손실 감소 |

## 7. 기존 범용 생성형 AI 사용 방식과의 차이

| 구분 | 범용 생성형 AI 대화 | 본 서비스 MVP |
|---|---|---|
| 사용자 경력 | 대화마다 다시 제공, 문맥 소실 가능 | 사용자 확인을 거친 확정 경력만 기준정보로 재사용 |
| 문서 추출 | 요약 결과가 곧 사실처럼 사용될 수 있음 | 추출 후보와 확정 정보를 명시적으로 분리 |
| 적합성 | 전체적인 인상이나 점수 중심이 되기 쉬움 | 요구사항별 상태와 사용자 경험 근거를 제시 |
| 회사 정보 | 출처·발행 시점이 빠지거나 최신성이 불명확할 수 있음 | 공식 출처 우선, 출처와 발행일/확인일 저장 |
| 시점 구분 | 일반 정보와 최근 정보가 혼재 | 회사 일반·최신·공고 시점 정보 구분 |
| 반복 분석 | 같은 입력에도 구조와 판단 기준이 달라질 수 있음 | 통제된 Workflow와 저장된 입력·결과 사용 |
| 후속 실행 | 답변이 대화에 머무름 | 지원 상태, 준비 전략, 문항별 근거로 연결 |
| 합격 판단 | 근거 없는 확률 생성 가능 | 합격 확률을 제공하지 않고 판단 근거와 불확실성 표시 |

차별점은 더 긴 답변이나 더 많은 에이전트가 아니라 **검증된 사용자 사실, 출처가 있는 외부 사실, 일관된 분석 절차, 저장 가능한 실행 결과**다.

## 8. MVP 기능 범위

### 8.1 반드시 포함할 기능(Must)

1. **사용자 문서 등록**
   - 이력서 또는 경력기술서 파일 등록
   - MVP 지원 형식은 우선 PDF로 제한하고 확대 가능성을 열어둠

2. **경력 정보 추출·검토·확정**
   - 경력, 프로젝트, 기술, 역할, 문제, 행동, 성과 추출
   - 항목별 원문 근거 표시
   - 수정·삭제·확정 지원
   - 미확정 정보는 공고 분석 근거에서 제외

3. **채용공고 본문 등록 및 직무 선택**
   - 공고 본문 직접 입력
   - 여러 직무가 포함된 경우 분석 대상 직무를 사용자가 선택 또는 범위 지정

4. **채용공고 구조화**
   - 주요 업무, 필수 조건, 우대 조건, 경력 요건, 기술, 고용형태, 근무지 추출
   - 추출 결과 확인·수정 지원

5. **요구사항별 경험 매칭과 적합성 분석**
   - PostgreSQL + pgvector 기반 사용자 경력 검색 우선 고려
   - 충족 / 부분 충족 / 확인 불가 / 미충족 상태
   - 필수·우대·직무 경험·기술·연차·도메인·근무 조건별 근거 제시
   - 강점, 부족한 역량, 추가 확인 사항, 지원 판단 근거 제공

#### 적합성 상태의 기획 정의

상태는 합격 가능성이나 사용자 역량의 절대 평가가 아니라, **선택한 공고의 개별 요구사항과 사용자가 확정한 정보 사이의 근거 관계**를 나타낸다.

| 상태 | 의미 | 적용 기준 | 예시 |
|---|---|---|---|
| 충족 | 요구사항을 만족한다고 볼 직접적이고 구체적인 확정 근거가 있다. | 기술명 일치뿐 아니라 역할·사용 맥락·기간 등 요구 수준을 판단할 근거가 충분해야 한다. | `Java/Spring Boot 서버 개발 경험`에 대해 확정된 실무 프로젝트와 담당 역할이 확인됨 |
| 부분 충족 | 일부 요소 또는 유사·전이 가능한 경험은 확인되지만 범위·수준·기간·도메인 중 일부가 부족하거나 직접 일치하지 않는다. | 확인된 부분과 부족한 부분을 각각 표시한다. 단순 키워드 유사도만으로 부여하지 않는다. | REST API 경험은 있으나 공고가 요구한 특정 도메인 경험은 없음 |
| 확인 불가 | 현재 확정된 사용자 정보만으로 충족 여부를 판단할 수 없다. | 문서에 언급이 없거나 근거가 모호한 경우의 기본 상태다. 추가 경험 확인을 요청할 수 있다. | Swagger 경험이 문서에 없어 보유 여부를 알 수 없음 |
| 미충족 | 요구사항을 충족하지 못한다는 명시적 근거가 있거나 객관적으로 비교 가능한 필수 기준에 도달하지 못한다. | 문서에 없다는 이유만으로 사용하지 않는다. 연차·자격·근무 가능 여부처럼 반대 사실이 확인될 때 사용한다. | 필수 경력 5년인데 확정 경력이 2년 3개월이고 다른 인정 경력이 없다고 사용자가 확인함 |

동일 요구사항에 복수 근거가 있으면 가장 유리한 키워드 하나만으로 상태를 정하지 않고 전체 근거를 함께 본다. AI가 상태를 제안하되 사용자는 근거를 확인하고 수정할 수 있어야 한다.

6. **요청 기반 회사 공식 정보 조사**
   - 회사가 직접 운영하거나 발행한 공식 홈페이지, 회사 소개, 채용 페이지, 기술 블로그, 뉴스룸·보도자료·IR 자료 중심
   - 제3자 기사·블로그·커뮤니티는 MVP 수집 대상에서 제외
   - 공고 1건당 최종 수집·저장 자료는 회사 소개·채용 페이지·최근 공식 자료를 합쳐 최대 5건
   - 출처 URL, 제목, 발행일 또는 확인일 저장
   - 일반 정보 / 최신 정보 / 공고 시점 정보 구분

7. **지원 근거와 준비 전략 생성**
   - 회사·직무 특성과 사용자 경험의 연결 근거
   - 지원 전 확인할 항목과 현실적인 준비 행동

8. **분석 결과 저장 및 지원 상태 관리**
   - 입력, 분석 시점, 근거, 출처 저장
   - 관심 / 지원 검토 / 지원 예정 / 지원 완료 / 보류 / 지원하지 않음 상태 관리

### 8.2 제한적으로 포함할 기능(Should/Could)

- 채용공고 URL 가져오기: 기술적·정책적으로 지원 가능한 사이트만 보조 기능으로 제공한다. 본문 붙여넣기가 기본 경로다.
- 포트폴리오 추가 등록: 이력서·경력기술서 흐름이 안정된 뒤 동일한 추출·병합 정책으로 확장한다.
- 분석 재실행: 사용자 경력 확정 내용이나 공고 내용이 변경된 경우 명시적으로 재실행한다.
- 자기소개서 문항 분석 및 근거 추천: 핵심 폐쇄 루프 검증 후 구현한다. 문항 유형 분류와 확정 경험·공식 회사 정보 근거 추천까지만 고려하며, 완성 답안 자동 제출은 제외한다.

### 8.3 MVP 범위 경계

- 회사 조사는 공고별 사용자 요청 시 실행하며 상시 수집하지 않는다. 공식 출처만을 대상으로 하고 공고 1건당 최대 5건을 채택하면 종료한다.
- 검색 결과 전체를 별도 지식베이스로 축적하는 기능은 MVP에서 제외한다.
- 분석 결과는 의사결정 보조이며 채용 자격 충족을 법적·공식적으로 판정하지 않는다.
- AI는 의미 추출·분류·연결 설명을 담당하고, 상태 전이·필수값 검증·날짜 비교·저장·중복 방지는 일반 애플리케이션 로직이 담당한다.

## 11. MVP 검증 방법

### 11.1 단계 1: 골든 샘플 오프라인 검증

첨부자료 5개를 고정 검증 세트로 사용한다.

1. S1~S3에서 동일 프로젝트가 중복 생성되지 않고 병합 후보로 제시되는지 확인한다.
2. 기간, 기술, 역할, 문제, 행동, 성과가 원문 근거와 함께 추출되는지 확인한다.
3. S4의 복수 직무 표에서 사용자가 선택한 직무만 분리되는지 확인한다.
4. S5 CRM 직무의 업무·자격·우대·고용형태·근무지를 구분하는지 확인한다.
5. S5 요구사항과 확정 경력의 연결 결과를 사람이 직접 판정한 기준안과 비교한다.
6. 첨부자료에 없는 제약 CRM·SFA 경험 등을 생성하지 않는지 확인한다.

### 11.2 단계 2: 대표 과업 사용성 검증

3~5명의 취업 준비자에게 자신의 문서와 실제 관심 공고 2개 이상으로 다음 과업을 수행하게 한다.

- 문서 등록 후 추출 결과 검토·확정
- 공고 등록 후 요구사항별 근거 확인
- 지원/보류/지원하지 않음 결정
- 회사 근거 1개 이상 선택
- 지원 상태 변경

과업 완료 시간, 수정 횟수, 잘못된 사실, 지원 판단 완료 여부, 유용성 평가와 자유 의견을 수집한다. 자기소개서 문항 분석은 Should 기능이 구현된 뒤 별도 과업으로 검증한다.

### 11.3 단계 3: 기준선 비교

동일 공고에 대해 다음 두 방식을 비교한다.

- 기존 방식: 범용 AI 새 대화 + 수동 회사 검색
- MVP 방식: 확정 경력 + 통제된 공고 분석 + 공식 출처 검색

비교 항목은 총 소요 시간, 다시 입력한 정보량, 근거 추적 가능성, 잘못된 경험 생성, 최종 판단 확신도다. 사용자 수가 적은 초기에는 통계적 우월성을 주장하지 않고 사례별 차이와 반복되는 문제를 확인한다.

## 12. 성공 지표

### 12.1 초기 성공 판단 기준

초기 표본은 첨부자료와 3~5명의 사용자로 작으므로, 정확도처럼 보이는 세밀한 백분율을 제품 성과로 주장하지 않는다. 공통 평가표로 사례별 결과를 기록하고 아래 조건을 중심으로 다음 개발 단계 진행 여부를 판단한다.

| 관찰 영역 | 기록 방법 | 초기 성공 판단 |
|---|---|---|
| 골든 샘플 오류 유형 | 누락, 잘못된 병합, 잘못된 분리, 근거 불일치, 상태 오판, 회사 정보 시점 오류를 건별 기록 | 반복되는 치명 오류 유형이 식별되고, 수정 후 동일 유형이 재현되지 않는지 확인할 수 있음 |
| 허위 경험 생성 | 확정 경력에 없는 경험을 보유 사실처럼 제시한 건을 별도 기록 | 골든 샘플과 사용자 검증에서 0건. 발생하면 다른 지표와 관계없이 출시 차단 오류로 취급 |
| 사용자 수정 횟수 | 경력 추출·공고 구조화·매칭 상태별 수정 건수와 수정 이유 기록 | 사용자가 결과를 포기하지 않고 확정할 수 있으며, 반복 테스트에서 동일 원인의 수정이 줄어듦 |
| 두 번째 공고부터의 시간 절감 | 첫 공고, 두 번째 이후 공고, 기존 수동 방식의 시작·종료 시간을 사용자별로 기록 | 두 번째 공고부터 경력 재입력·재정리 단계가 줄고, 다수 사용자 사례에서 실제 소요 시간이 단축됨 |
| 지원 판단 완료 여부 | 분석 후 `지원 예정`, `보류`, `지원하지 않음` 중 하나와 그 이유를 남겼는지 기록 | 사용자가 근거를 확인한 뒤 공고별 판단과 다음 행동을 완료할 수 있음 |
| 근거 추적 가능성 | 추출 경험, 요구사항 매칭, 회사 정보가 열 수 있는 원문·공식 출처와 연결되는지 점검 | 판단에 사용된 모든 핵심 사실의 출처를 사용자가 확인할 수 있음 |
| 회사 조사 범위 준수 | 공식 여부, 자료 유형, 발행일/확인일, 공고별 채택 건수 기록 | 제3자 자료 없이 최대 5건 안에서 필요한 근거를 확보하거나, 부족하면 검색 실패·근거 부족을 명시 |

초기 성공은 모든 사용자가 같은 결과를 내는 것이 아니라, **치명적인 사실 오류 없이 지원 판단을 끝내고 두 번째 공고부터 반복 작업이 줄어드는 사례가 일관되게 관찰되는지**로 판단한다. 정량 목표치는 오류 로그와 사용성 테스트가 축적된 후 베이스라인을 기준으로 설정한다.

### 12.2 허영 지표로 보지 않을 항목

- 생성된 문장 수
- 분석 결과의 길이
- 등록 공고 수만으로 본 활성도
- 모델이 산출한 단일 적합도 점수
- 근거 없는 합격 가능성 백분율

## 13. 주요 위험 요소와 대응 방향

| 위험 | 영향 | MVP 대응 방향 |
|---|---|---|
| PDF 추출 실패 또는 표 구조 손실 | 경력·공고 정보 누락 | 지원 형식을 제한하고 원문 미리보기·수동 수정·본문 입력 대체 경로 제공 |
| 동일 경험의 중복·충돌 | 잘못된 경력 기준정보 생성 | 자동 병합 대신 병합 후보 제시, 출처별 값 보존, 사용자 확정 필요 |
| 성과 수치·역할 과장 | 지원서 신뢰 훼손 | 수치와 역할에 원문 근거 표시, 미확정 상태 유지, AI의 보정 금지 |
| 임베딩 유사도 과신 | 관련 없는 경험 연결 | 벡터 검색은 후보 검색에만 사용하고 필수값·연차 등 규칙 검증 및 근거 기반 판단 병행 |
| `미충족` 오판 | 지원 기회 상실 | 문서에 없으면 기본적으로 확인 불가로 두고 명시적 반대 근거가 있을 때만 미충족 사용 |
| 공고 내 복수 직무 혼합 | 잘못된 요구사항 분석 | 직무 선택 단계를 두고 선택 범위 확인 후 분석 |
| 외부 검색의 부정확성·최신성 | 잘못된 지원동기 생성 | 공식 출처 우선, 출처·발행일/확인일 필수, 검색 실패를 명시 |
| 채용 사이트 접근 제한 | URL 입력 기능 불안정 | 본문 붙여넣기를 기본 경로로 설계하고 URL은 지원 사이트만 보조 제공 |
| 회사 조사 범위 확대 | 비용·지연·운영 복잡도 증가 | 사용자 요청 시 공고 단위 실행, 공식 출처만 사용, 공고당 최대 5건 채택 후 종료 |
| LLM 비용과 응답 지연 | 개인 운영 부담 | 단계별 호출, 입력 재사용, 변경 없을 때 결과 재사용, 모델 등급화는 구현 직전 결정 |
| 민감한 개인정보 저장 | 보안·신뢰 문제 | 최소 수집, 문서 삭제와 데이터 삭제 제공, 보관 정책·마스킹 범위는 제품 정책으로 확정 |
| 결과의 결정론 부족 | 재분석 시 판단 변화 | Workflow 입력·출력 스키마, 모델·프롬프트 버전, 분석 시점 저장 |
| 과도한 초기 설계 | 개발 착수 지연 | 본 기획서 이후 필요한 5개 문서만 순차 작성하고 세부 AI/API 문서는 구현 직전에 작성 |

## 14. 향후 확장 방향

MVP 검증 후 실제 사용 빈도와 요청을 근거로 다음 순서를 검토한다.

1. DOCX·텍스트·이미지 문서 입력 확대와 포트폴리오 병합 정교화
2. 지원 가능한 채용 사이트의 URL 가져오기 확대
3. 경력 버전 관리와 여러 이력서 버전 비교
4. 공고별 이력서·경력기술서 보완 제안
5. 자기소개서 초안 작성 보조 및 사용자 편집 이력 관리(자동 제출 제외)
6. 면접 질문과 답변 근거 카드 생성
7. 사용자 동의 기반 GitHub 프로젝트 근거 가져오기
8. Calendar 일정 등록 등 외부 도구 연동이 필요해질 때 MCP 적용 검토
9. 직무별 요구사항 분류 체계와 평가 데이터셋 확대
10. 개인 데이터 보호·삭제·내보내기 기능 강화
11. IT·개발 직군에서 검증한 범용 경험·요구사항 구조를 바탕으로 기획, 데이터, 인프라, 보안, 영업, 연구 등 다른 직무 분류 체계 확장

확장 여부는 “기술적으로 흥미로운가”가 아니라 반복 사용, 시간 절감, 오류 감소에 기여하는지로 결정한다.

## 15. 단계 종료 점검

### 15.1 앞 문서 또는 프로젝트 원칙과 충돌하는 내용

- 현재 앞 문서는 없으며, 제시된 프로젝트 원칙과 직접 충돌하는 내용은 없다.
- v0.2에서는 회사 공식 정보 조사는 공고당 최대 5건의 Must로 제한하고, 자기소개서 문항 분석은 Should로 이동하여 범위 충돌을 해소했다.

### 15.2 개인 개발자 MVP로 범위가 과도한 내용

- 서로 다른 PDF 레이아웃을 범용적으로 완벽하게 파싱하는 것
- 모든 공고 URL을 자동으로 가져오는 것
- 회사의 일반·최신·공고 시점 정보를 완전히 자동 분류하는 것
- 사용자 경력과 요구사항의 상태를 IT·개발 이외 모든 직무에서도 높은 정확도로 자동 판정하는 것
- 자기소개서 문항 분석까지 첫 배포에 포함하는 것

따라서 첫 구현에서는 PDF 입력 유형과 직무 선택 방식을 제한하고, 회사 조사는 공식 자료 최대 5건으로 제한한다. 자기소개서 문항 유형은 Should 기능 구현 직전에 정한다.

### 15.3 아직 결정되지 않은 사항

- MVP가 지원할 최초 문서 형식: PDF만 지원할지, 텍스트 붙여넣기를 함께 지원할지
- 포트폴리오 입력을 첫 출시부터 받을지 후속으로 미룰지
- 하나의 사용자 계정에서 여러 경력 프로필 또는 이력서 버전을 지원할지
- 공고 구조화 결과도 사용자 확정을 필수로 할지, 수정 가능 상태만 둘지
- `미충족` 판정이 가능한 명시적 조건의 범위
- 적합성의 종합 요약을 등급으로 제공할지, 요구사항별 결과만 제공할지
- 초기 UI 범위와 인증 방식
- LLM·임베딩 모델, 검색 도구, 운영 비용 상한

### 15.4 사용자 확인이 필요한 제품 정책

1. 사용자가 확정한 경력만 분석에 사용할지, 미확정 정보도 경고 표시 후 후보로 검색할지
2. 문서 원본과 추출 데이터를 각각 얼마나 오래 보관할지
3. 문서 삭제 시 분석 결과와 근거도 함께 삭제할지
4. 공고가 삭제·변경된 뒤 저장된 원문과 분석 결과를 유지할지
5. 종합 지원 판단의 표현을 `추천 / 조건부 추천 / 보류`처럼 제공할지
6. Should 단계의 자기소개서 기능을 근거 추천까지만 둘지 짧은 개요까지 제공할지

### 15.5 이후 단계에서 구체화할 사항

- 2단계: 사용자 여정별 시작·종료 조건, 예외 흐름, 사용자 행동
- 3단계: 기능·비기능 요구사항 ID, 우선순위, 완료 조건
- 4단계: 구현 순서, 스토리 분할, MVP 릴리스 기준
- 5단계: 사용자 경력, 문서 근거, 공고, 요구사항, 매칭, 회사 출처, 지원 상태의 도메인·데이터 초안
- 6단계: Java 21·Spring Boot·PostgreSQL/pgvector 중심 초기 아키텍처와 외부 AI·검색 경계
- 각 기능 구현 직전: AI/RAG/검색 Workflow, API 명세, 테스트 및 AI 품질 평가 계획

## 16. 기획 단계 결론

MVP의 핵심은 “AI가 지원서를 대신 써주는 것”이 아니라 다음 폐쇄 루프를 검증하는 것이다.

> 문서에서 경력 후보 추출 → 사용자 확인·확정 → 공고 요구사항 구조화 → 확정 경력 검색·연결 → 공식 회사 근거 결합 → 지원 판단·준비 행동 저장

첫 번째 성공 조건은 기능 수가 아니라, 첨부된 실제 사례에서 **없는 경험을 만들지 않고**, **요구사항마다 확인 가능한 근거를 제시하며**, **두 번째 공고부터 반복 작업을 실제로 줄이는 것**이다.
