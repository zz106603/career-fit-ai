## 1. 문서 목적과 범위

본 문서는 확정된 기획과 유스케이스를 구현·테스트 가능한 요구사항으로 변환한다. 핵심 폐쇄 루프는 `경력 PDF 등록 → 추출 후보 확인·확정 → 공고 등록·구조화 → 공식 회사 정보 조사 → 요구사항별 경력 매칭 → 판단·전략 자동 저장 → 지원 상태 관리`다.

API URI·DTO, 테이블·컬럼, 클래스·패키지, 구체 모델·검색 제품·프롬프트는 확정하지 않는다. 자유형 Multi-Agent와 서비스 내부 MCP 분리는 범위에 포함하지 않는다.

### 1.1 우선순위 기준

| 등급 | 기준 |
|---|---|
| Must | 핵심 사용자 폐쇄 루프와 그 안전한 복구에 필수 |
| Should | 핵심 흐름 없이도 동작하지만 사용성·품질을 크게 높임 |
| Could | 검증 이후 확장 후보 |
| Won’t | 현재 MVP에서 명시적으로 제외 |

### 1.2 공통 인수 원칙

- 모든 인수 조건은 독립 테스트가 가능한 Given / When / Then으로 해석한다.
- AI 추출 후보는 사용자 확정 전까지 분석 근거가 아니다.
- 문서에 근거가 없으면 기본적으로 `확인 불가`이며, 명시적 충돌만 `미충족`이다.
- 회사 조사 실패는 경력 매칭과 판정을 중단시키지 않는다.
- 완료 또는 부분 완료 결과는 별도 저장 동작 없이 자동 보존한다.

## 2. 용어 정의

| 용어 | 정의 |
|---|---|
| 경력 문서 | 사용자가 등록한 PDF 이력서 또는 경력기술서 |
| 추출 후보 | AI가 원문에서 식별했으나 사용자가 아직 확정하지 않은 경력 정보 |
| 확정 경력 | 사용자가 확인·수정 후 명시적으로 확정한 경험 정보 |
| 경험 | 경력, 프로젝트, 기술, 역할, 문제, 행동, 성과를 포함하는 재사용 가능한 근거 단위 |
| 원문 근거 | 문서명, PDF 페이지 번호, 원문 발췌 또는 사용자 직접 입력 표시 |
| 공고 구조 | 회사명, 직무명, 주요 업무, 필수·우대 조건, 연차, 기술, 근무 조건의 구조화 결과 |
| 이상 징후 | 복수 직무, 회사명 불명확, 타 직무 요건 혼입 가능성, 필수·우대 분류 불확실성 |
| 공식 출처 | 회사가 직접 운영하거나 발행한 홈페이지, 채용 페이지, 기술 블로그, 뉴스룸, 보도자료, IR 자료 |
| 분석 실행 | 특정 공고 구조와 확정 경력 스냅샷을 입력으로 한 Workflow 1회 |
| 완료 | 필수 분석 산출물이 생성·저장된 상태. 회사 정보 제한사항은 있을 수 있음 |
| 부분 완료 | 요구사항별 판정·근거는 저장됐으나 최종 지원 전략 등 후속 산출물이 실패해 제한된 결과만 제공되는 상태 |
| 충족 | 요구 수준을 만족하는 직접적·구체적 확정 근거가 있음 |
| 부분 충족 | 일부 요소나 전이 가능한 근거는 있으나 범위·수준·기간·도메인 일부가 부족함 |
| 확인 불가 | 현재 확정 정보만으로 충족 여부를 판단할 수 없음 |
| 미충족 | 사용자 확정 정보와 공고 조건이 명시적으로 충돌함 |
| 지원 상태 | 관심, 지원 검토, 지원 예정, 지원 완료, 보류, 지원하지 않음 중 하나 |

## 3. 사용자 역할

| 역할 | 설명 | 권한 범위 |
|---|---|---|
| 비회원 | 회원가입·로그인 전 사용자 | 인증 화면과 공개 안내만 접근 |
| 회원 | 자신의 경력·공고·분석·지원 상태를 관리하는 사용자 | 본인 데이터만 생성·조회·수정·삭제 |
| 시스템 작업자 | 비동기 문서·분석 Workflow를 수행하는 내부 주체 | 요청 소유자의 범위 안에서만 처리하며 사용자 역할로 노출하지 않음 |

운영자용 사용자 문서 열람·수정 화면은 MVP 요구사항에 포함하지 않는다.

## 6. 데이터 및 근거 관리 요구사항

1. 후보 경력과 확정 경력은 상태와 확정 시점으로 구분한다.
2. 공고 분석은 실행 당시의 확정 경력과 공고 구조 스냅샷을 사용한다.
3. PDF 근거는 문서명·페이지 번호·원문 발췌를 제공한다. 좌표 기반 정밀 하이라이트는 제외한다.
4. 사용자 직접 입력 경력·대체 텍스트는 PDF 근거와 혼동되지 않도록 출처 유형을 표시한다.
   - 직접 입력 경험은 `경험명 또는 프로젝트명`과 `역할 또는 수행 내용`을 필수로 한다.
   - 사용자 명시적 확정 후 분석 근거로 사용할 수 있다.
   - 시스템은 사용자가 직접 입력한 사실의 진위를 검증·보증하지 않는다.
5. 중복 경험 후보와 사용자의 병합·분리 결정 보존은 Should다. Must에서는 경험을 각각 독립 항목으로 관리할 수 있다.
6. 판정에는 요구사항, 상태, 설명, 연결 경험, 근거, 실행 버전을 연결한다.
7. 회사 정보에는 공식 출처 URL, 제목, 발행일(있는 경우), 확인일을 저장한다. 일반·최신·공고 시점 정보의 완전 자동 분류는 Should다.
8. 완료·부분 완료 분석의 입력, 단계 상태, 결과, 실패, 근거, 출처를 하나의 실행 맥락으로 재조회할 수 있어야 한다.
9. 확정 경력 수정이나 재분석은 과거 결과를 덮어쓰지 않는다.
10. 보관·삭제·연쇄 삭제 정책은 구현 전 확정한다.

### 7.2 AI와 일반 로직의 책임

- AI: 비정형 텍스트의 후보 추출, 의미 분류, 관련성 설명, 근거 범위 내 요약·전략 생성.
- 일반 로직: 인증·인가, 파일·필수값·상한 검증, 사용자 격리, 상태 전이, 중복 실행 제어, 날짜·명시 연차 비교, 출처 수 상한, 저장.
- 벡터 검색은 관련 경험 후보 탐색에만 사용하며 판정 자체를 유사도 점수로 결정하지 않는다.
- 공식 검색 결과는 도메인·발행 주체를 검증하며 제3자 기사·커뮤니티로 자동 대체하지 않는다.
- 자유형 Multi-Agent와 내부 MCP 분리는 사용하지 않는다.

## 8. 상태 및 실패 처리 요구사항

### 8.1 상태 원칙

- 화면에는 실시간 백분율이 아닌 단계 상태를 표시한다.
- 문서 분석, 회사 조사, 공고 분석은 서로 구분된 실행 상태를 가진다.
- 회사 조사 실패만 있고 핵심 판정·전략이 생성되면 `완료(회사 지원 근거 제한)`이다.
- 요구사항별 판정이 저장됐으나 최종 지원 전략 등 후속 산출물이 누락되면 `부분 완료`다.
- 개별 요구사항별 독립 실패 복구와 성공 산출물의 선택적 재사용은 MVP Must가 아니다.
- 저장 성공 전에는 완료·부분 완료로 최종 노출하지 않는다.
- 정확한 상태명과 허용 전이는 도메인·데이터 설계에서 확정한다.

### 8.2 재실행 단위

| 단위 | MVP 처리 | 우선순위 |
|---|---|---|
| 경력 문서 분석 전체 재실행 | 입력부터 후보 생성까지 새 실행 | Must |
| 회사 공식 정보 조사 재실행 | 조사만 새 실행, 시점 구분 | Should |
| 공고 분석 전체 재실행 | 입력 검증부터 자동 저장까지 새 실행 | Must |
| 개별 요구사항별 독립 실패 복구·성공 산출물 선택 재사용 | MVP 미구현 | Should |

### 8.3 실패 분류

- 입력 오류: 사용자 수정 전 재시도하지 않는다.
- 일시적 외부 오류: 제한된 자동 재시도 후 실패 상태로 전환한다.
- 회사 식별·검색 실패: 분석을 계속하고 제한사항을 표시한다.
- 최종 전략 생성 실패: 요구사항별 판정이 저장됐다면 제한된 부분 완료 결과를 자동 저장한다.
- 개별 요구사항 처리 실패: MVP에서는 공고 분석 전체 재실행으로 복구한다. 성공 항목만 선택적으로 재사용하는 기능은 Should다.
- 저장 실패: 완료로 표시하지 않고 안전한 제한 재시도를 수행한다.
- 동일 실행 중복 요청: 기존 진행 실행을 반환하거나 중복 요청임을 알린다.

## 9. 보안·개인정보 요구사항

1. 인증정보는 평문 저장하지 않으며 전송 구간 보호를 전제로 한다.
2. 원본 문서, 추출·확정 경력, 공고, 분석, 검색 인덱스 모두 사용자 소유 범위를 적용한다.
3. 파일 다운로드·미리보기·비동기 작업 결과에도 동일한 접근 제어를 적용한다.
4. 로그에 비밀번호·토큰·문서 전체 원문·LLM 전체 요청·응답을 기본 기록하지 않는다.
5. 오류 메시지는 타 계정·자원의 존재나 내부 비밀정보를 노출하지 않는다.
6. 외부 AI·검색으로 보내는 개인정보는 목적 달성에 필요한 최소 범위로 제한한다.
7. 원본 문서·추출 데이터·분석 결과의 보관 기간과 사용자 삭제 범위를 구현 전에 확정한다.
8. 운영자 접근과 장애 분석용 데이터 열람 정책은 MVP 운영 전 별도 결정한다.
9. 가입·로그인의 구체 방식, 세션·토큰 선택과 인증 수단은 시스템 아키텍처에서 결정한다. 사용자별 데이터 격리와 보호 자원의 소유권 검사는 방식과 무관하게 Must다.
10. 비밀번호 재설정, 소셜 로그인, 복잡한 계정 복구는 1차 MVP에서 구현하지 않으며 후속 Should로 관리한다.

## 12. 미결정 사항과 후속 설계 항목

### 12.1 기획서 v0.2 또는 유스케이스 v0.2와 충돌하는 요구사항

기준 문서 간 표현 차이와 이번 우선순위 조정을 아래처럼 관리한다. **구현 정책은 가장 최근 확정 문서인 본 요구사항 명세서 v0.2를 우선한다.**

1. 기획서 8.1·8.3의 회사 조사가 `요청 기반`, `사용자 요청 시 실행`으로 적혀 있으나 유스케이스 v0.2는 공고 분석에 기본 포함한다. 본 명세는 기본 포함으로 정의한다.
2. 기획서 적합성 정의 말미의 `사용자는 근거를 확인하고 수정` 표현은 판정값 수정으로 읽힐 수 있으나, 유스케이스 v0.2는 직접 덮어쓰기를 제외한다. 본 명세는 이의 피드백·경력 보완·전체 재분석만 허용한다.
3. 기획서 시나리오 E의 `사용자가 분석 결과를 저장` 표현과 달리 유스케이스 v0.2는 완료·부분 완료 자동 저장이다. 본 명세는 자동 저장을 적용한다.
4. 기획서 단계 종료점검의 일부 미결정 사항(PDF 대체 텍스트, 공고 구조화 확인 정책)은 유스케이스 v0.2에서 이미 확정됐다. 본 명세는 PDF 실패 시 붙여넣기와 이상 징후 시에만 필수 확인을 적용한다.
5. 유스케이스 v0.2는 회사 조사 단독 재실행을 MVP 재실행 단위로 제시하지만, 본 명세 v0.2는 1차 구현량을 줄이기 위해 이를 Should로 내린다. Must 복구 경로는 공고 분석 전체 재실행이다.
6. 유스케이스 v0.2의 AI 판정 이의 기록은 본 명세에서 Should다. 누락 경력의 추가·수정·명시적 확정과 공고 분석 전체 재실행은 Must로 유지한다.
7. 유스케이스 v0.2는 일부 요구사항 실패의 부분 완료를 폭넓게 표현하지만, 본 명세의 Must는 `요구사항별 판정 저장 성공 + 최종 전략 생성 실패`의 제한 결과 조회까지다. 항목별 독립 복구와 성공 산출물 재사용은 Should다.

#### 다음 기획서 개정 시 수정할 표현

- 회사 조사를 `사용자 요청 시 실행`이 아니라 공고 분석 기본 포함으로 수정한다.
- `사용자가 분석 결과를 저장`을 완료·부분 완료 자동 보존으로 수정한다.
- `사용자가 판정을 수정`으로 읽힐 수 있는 문장을 이의 피드백 또는 경력 보완 후 재분석으로 수정한다.
- 직접 입력 경력의 최소 필수값, `사용자 직접 입력` 출처, 사실 검증 비보증 원칙을 추가한다.
- 회사 정보 Must 범위를 공식 자료 합계 최대 5건과 출처·날짜 저장으로 축소하고 완전 자동 시점 분류를 후순위로 표시한다.
- 지원 상태 Must를 현재 상태 저장·목록 표시로 제한하고 상세 전이·이력·사유를 후순위로 표시한다.
- 회사 조사 단독 재실행과 개별 요구사항 복구의 최신 우선순위를 본 명세와 동기화한다.

### 12.2 Must로 지정하기에 범위가 과도한 요구사항

- 일반·최신·공고 시점 회사 정보의 완전 자동 분류는 Must로 보장하기 과도하다. MVP Must는 공식 소개·채용·인재상·직무 관련 최근 자료를 합계 최대 5건 수집하고 출처와 날짜를 저장하는 수준이다.
- 서로 다른 PDF 레이아웃의 완전한 파싱과 동일 경험 자동 병합은 Must가 아니다. PDF 기본 추출만 Must이고 유사 경험 후보 표시는 Should다.
- 회사 공식 도메인의 완전 자동 식별은 과도하다. 실패·모호성 상태와 분석 지속이 Must다.
- 모든 AI 결과의 의미적 정확성을 일괄 수치 SLA로 보장하는 것은 초기 표본에서 과도하다. 골든 샘플 오류 유형과 허위 경험 0건을 우선 평가한다.

### 12.3 Should 또는 Could로 내려야 하는 요구사항

- 자기소개서 문항 분석·근거 추천, 공고 URL 가져오기, 포트폴리오 등록: Should.
- 유사 경험·동일 공고 중복 감지, 회사 조사 단독 재실행, AI 판정 이의 피드백: Should.
- 개별 요구사항별 독립 복구·성공 산출물 선택 재사용, PDF 좌표 하이라이트, 상세 분석 비교: Should.
- 회사 정보 시점의 완전 자동 분류와 지원 상태 상세 전이·전체 이력 UI·사유 입력: Should.
- 사용자 판정과 AI 판정의 별도 값 관리·비교: Could.
- 비밀번호 재설정, 소셜 로그인, 지원 상태 변경 전체 이력 UI는 폐쇄 루프 이후 Should 후보.
- 일반 조회 p95 1초 목표는 초기 운영 기준이 없어 Should로 두고 실제 측정 후 조정한다.

### 12.4 아직 결정되지 않은 제품 정책

1. PDF 최대 용량·페이지 수, 암호화 PDF 거절 범위.
2. 공고 분석이 가능한 최소 본문 기준.
3. 동일 공고 등록 시 기존 공고 갱신과 새 공고 생성의 선택 규칙.
4. 사용자 직접 입력 경력은 `경험명 또는 프로젝트명`과 `역할 또는 수행 내용`을 최소값으로 확정했다. 추가 검증·경고 수준은 미결정이다.
5. 종합 지원 판단을 문장만 제공할지 제한된 등급도 제공할지.
6. 보류·지원하지 않음 사유 입력은 Should로 확정했다. 후속 버전에서 선택 또는 필수 여부를 결정한다.
7. 원본 문서·추출 데이터·분석 결과의 보관 기간과 삭제 범위.
8. 자동·수동 재시도 횟수, 호출 비용 상한과 사용자 노출 방식.
9. `최근 공식 자료`의 기간 기준과 자료 유형별 우선순위.
10. 인증 방식, 계정 복구와 탈퇴 범위.

## 15. 단계 결론

본 명세의 MVP 완료 기준은 등록 기능의 개수가 아니라 다음 폐쇄 루프가 한 사용자 데이터 범위에서 안전하게 끝나는 것이다.

> PDF 원본과 근거가 보존된 후보 추출 → 사용자 확정 → 조건부 공고 확인 → 공식 회사 조사 실패를 허용하는 요구사항 판정 → 근거·제한사항이 있는 결과 자동 저장 → 지원 상태 결정 → 경력 보완 후 전체 재분석

도메인·데이터 설계로 이동하기 전에 12.9의 입력 상한, 공고 최소 기준, 경험 경계·버전, 부분 완료 경계, 보관·삭제, 회사 출처 선별, 판단 표현 정책을 우선 결정해야 한다.
